# 章节树生成：目录提供章节骨架，LLM 负责切分知识点

PDF 目录的"章/节"层级与教学结构通常对齐稳定，但目录最细层级（如 3.2.1 子节）常表示数页内容，与"5-15 分钟知识点"的目标颗粒度不匹配。因此章节树生成策略统一为：从 PDF 目录抽取前两层作为 **Chapter** 与 **Section** 骨架，再由 LLM 在每个 Section 内独立重新切分 **KnowledgePoint**，目录的更深层级一律忽略。

## Considered Options

- **LLM 一次性生成全树（路径 A）**：把整份 PDF 文本丢给 LLM 输出完整 4 层 JSON。长输出截断风险高、同一份 PDF 多次生成结果不一致，答辩演示风险大。
- **目录直接映射到全部 3 层（B.1 / B.2）**：用目录的"章/节/小节"直接作为 `Chapter/Section/KP`。会出现"一个 KP 等于 30 分钟+ 内容"的颗粒度错位。
- **LLM 看目录决定层级映射（B.3）**：把"映射策略"也交给 LLM 推理，可控性回退到路径 A 的水平。
- **多阶段独立调用流水线（路径 C）**：分步识别章 → 节 → KP。累计延迟 1-3 分钟；错误传播严重；演示体验差。

## Consequences

- **PDF 必须有可解析的目录**（bookmark 或可识别的 TOC 页）。无目录的资料如何降级处理仍未定义，是本决策的未解依赖。
- LLM 切分 KP 时需要在 prompt 中显式约束"5-15 分钟内容量"，量化方式（字数？token 数？）尚未确定。
- 目录的小节层级（3 层及以上）信息会被丢弃。若用户期望"教材里的 3.2.1 直接对应一个 KP"，会感到 surprising —— UI 上需要提示用户"KP 是系统重新切分的，不是教材原小节"。
