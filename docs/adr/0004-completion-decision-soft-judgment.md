# 通过判定：系统给建议，用户决定下一步

每个 **KnowledgePoint** 的学习循环结束时，系统对学生表现做评估并给出建议（"我觉得你掌握了" / "我觉得你 X 还有薄弱"），但**不阻拦**学生进入下一个 KP。学生在"下一个 KP"和"再练一题/再讨论"之间自由选择。系统的建议被记录为薄弱点数据。

## Considered Options

- **系统硬判定（按评分阈值自动通过/不通过）**：LLM 评分不稳定，把对的判错会让学生卡死，是严重的 UX 风险，毕设答辩演示中一旦触发即翻车。
- **用户全权决定（评分只是个数字）**：把评分模块做成摆设，浪费教学价值，且让"通过"成为空概念。

## Consequences

- LLM 评分质量不再是"卡学生"的杀手级因素，但仍影响"建议质量" —— 评分错误率不再致命，但仍需关注。
- UI 必须区分两个独立动作："系统建议（通过/未通过）"和"用户决定（下一节/继续练）"。不能让用户误以为"点了下一节 = 系统认可我通过了"。
- 薄弱点池有两条来源：系统判定的薄弱（评分驱动）+ 用户强行跳过未掌握的（行为驱动）。两条来源进同一个池，不区分。
