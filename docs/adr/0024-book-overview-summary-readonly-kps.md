# 全书导读 / 全书总结：辅文剔除 + 总-分-总只读 KP

无目录 PDF / Markdown 走 LLM 骨架推断时，序言/前言/结语/后记/附录等**辅文**会被推断成与正文平级的章，再被 ADR-0012 的"每节强制切 3-7 个 KP"放大成一堆无意义 KP（各自生成物料/习题/对话/测评）。改为：**辅文不进正文树**，并在 body 首尾各注入一个合成的只读知识点——**全书导读**（开篇）与**全书总结**（收尾），形成"总—分—总"。这两个 KP 的物料**由全书章节大纲 + 辅文原文综合生成**（不是某页切片），**可与老师对话但不出题、不测评、不过关**，且不计入课程进度。

辅文识别只在章列表的**开头连续段**与**结尾连续段**按保守标题词命中（`序言/前言/引言/导言/结语/后记/跋/附录/致谢/参考文献/索引…`）；**中间章永不剔除**——避免把正文首章（如正经的"引言/导言"章）误判为辅文。

`KnowledgePoint.boundary` 是 JSONB，合成 KP 写 `{"kind": "overview"|"summary", "matter_pages": [[s,e]…]}` 且**不写 page_start/page_end**：既复用现有列（无需迁移），又让对话天然走全书语义检索（无页范围 → 走 embedding top-k，正好支撑"第3章和第5章怎么串"这类跨章提问）。物料**复用 `KPMaterialPayload` 形状**，故苏格拉底对话层零改动即可消费。

## Considered Options

- **直接剔除辅文、不补任何东西**：最省，但放弃了"开篇导读 / 收尾收束"的教学价值。否决——用户要的是"总-分-总"。
- **把序言/结语字面改名成 1 个 KP**：便宜，但这两个 KP 的源文本只是辅文原文（多为致谢/动机等元信息），作为"总"很空。否决。
- **（选定）综合生成 + 只读 + 可聊**：导读/总结物料由"全书大纲 + 辅文原文"综合而成，是真正的全书路线图/知识地图；学生可就它与老师对话，但不进 assessor/习题/过关闭环。
- 交互形态：纯静态面板 vs **可聊不测评** vs 保留机制自动过关 —— 选**可聊不测评**（保留苏格拉底层的价值，去掉无意义的过关）。
- 综合输入：仅大纲 vs **大纲+辅文原文** vs 大纲+各章正文抽样 —— 选**大纲+辅文原文**（辅文作者意图可入导读，成本可控）。
- 合成 KP 的树形：因 `KnowledgePoint` 外键强制挂在 `Section → Chapter` 下（4 层固定，见 ADR-0006/CONTEXT），物理上必然是"章→节→KP 三层同名"。**不改数据模型**（改模型属过度设计），改为**前端把"仅含单个合成 KP 的章"压成一行**渲染（保留「导读/总结」徽标）。

## Consequences

- **ADR-0001（章节树生成策略）**：在"骨架推断 → 每节切 KP"之间新增一道**辅文分流**，并在 body 外包裹两个合成章；正文生成路径不变。
- **ADR-0012（按数量+概念切 KP，每节 3-7）**：合成导读/总结 KP **豁免**该规则——它不是从某节切出来的，是全书级的单个 KP。
- **ADR-0004（完成判定软裁决）/ ADR-0006（章节树构建后不可变）**：合成 KP **没有完成概念**，永不 `passed`；后端对其 `/advance`、`/assessment`、`/exercise-set`、`/submissions` 一律 409（前端本就不展示入口，后端兜底）。ADR-0006 的"构建后结构不可变"仍成立——合成节点在构建时一次性确定。
- **ADR-0020（KPMaterial 形状）**：被复用而非扩展——导读/总结物料是同 `KPMaterialPayload` 的另一来源（`generate_book_overview_material` + `book_overview.md`），prewarm 按 `boundary.kind` 路由，不走 boundary 页生成器。
- **进度数学**：合成 KP 排除出课程"已过/总数"统计与章节树状态聚合，否则合成章永远卡 `in_progress`、课程永不到 100%。聊天仍会把它内部置 `in_progress`，但 rollup/进度视其为不计数。
- **辅文识别是启发式**：保守词表 + 仅首尾连续段，已知残余风险是"没有序言、正文首章正好叫'引言/导言'"会被误判；用户接受此代价（最懂教材，可后续调词表）。
- 物理结构是"章→节→KP 三层同名"，但 UI（CoursePage 主树 + KPPage 迷你树）压成一行——后来人勿因看到 DB 三层就去"修复"数据模型，这是外键约束下的有意取舍。
